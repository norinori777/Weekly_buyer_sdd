# Research: 商品登録画面の料理メニュー入力

## Decision 1: 料理メニューは日次の文脈情報として保存する
- Decision: 料理メニューは商品データとは分けて、選択中の日付と朝・昼・夜の区分にひも付く情報として保存する。
- Rationale: 商品一覧に混ざらず、日ごとの献立文脈として扱いやすくなる。
- Alternatives considered: 商品行にメモを付ける案。商品リストと混ざり、表示の責務が曖昧になるため採用しない。

## Decision 2: メニューは区分ごとに複数保持する
- Decision: 朝、昼、夜それぞれに複数件の料理メニューを持てるようにする。
- Rationale: 1食ごとに複数メニューを並べる通常の使い方に対応しやすい。
- Alternatives considered: 各区分を1件だけに限定する案。献立の幅が狭くなり、入力の自由度が下がる。

## Decision 3: 入力はセクションごとの追加ボタンから開く
- Decision: 朝・昼・夜それぞれのセクション右上に追加ボタンを置き、押した区分に対して下からスライドアップする入力フォームを開く。
- Rationale: 現在のセクション文脈に紐づいた操作として分かりやすく、画面遷移を増やさずに入力できる。
- Alternatives considered: 各セクション内に固定入力欄を置く案。画面が散らかり、編集導線が分散する。

## Decision 4: 空の区分は非表示にする
- Decision: その日に料理メニューがない区分は表示しない。
- Rationale: 画面が簡潔になり、必要な情報だけに集中できる。
- Alternatives considered: すべての区分を空状態付きで表示する案。毎回余白が多くなり、情報が散らかる。

## Decision 5: 検証は日次保存と画面表示に分ける
- Decision: 日付ごとの保存・再表示は repository テスト、候補表示・✖削除・非表示は widget テストで確認する。
- Rationale: 保存ロジックと画面ロジックを分けると、失敗箇所を特定しやすい。
- Alternatives considered: E2E のみで確認する案。初期段階では修正コストが高い。
