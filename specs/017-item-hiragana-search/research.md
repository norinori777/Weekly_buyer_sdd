# Research: 商品名のひらがな候補表示

## Decision 1: ひらがな列は nullable で追加する
- Decision: 商品マスタにひらがな列を追加し、既存行は null のまま残す。
- Rationale: 既存商品を壊さずに機能を導入でき、古いデータも商品名検索で継続利用できる。
- Alternatives considered: 非 null 必須化。全件入力が必要になり、移行負荷が高い。

## Decision 2: 設定画面ではひらがなを必須にする
- Decision: 商品の新規作成と編集では、商品名とひらがなを両方入力させる。
- Rationale: 候補表示の精度を安定させるには、編集後のデータは読み付きで揃っている必要がある。
- Alternatives considered: ひらがな任意化。候補の品質がばらつき、機能価値が下がる。

## Decision 3: 候補検索は商品名とひらがなの両方を対象にする
- Decision: 商品登録画面での候補表示は、入力文字列を商品名とひらがなの両方に対して部分一致させる。
- Rationale: 表記ゆれに強くなり、ユーザーは覚えている読みからでも商品を見つけられる。
- Alternatives considered: 商品名のみ検索。今回の要望を満たせない。

## Decision 4: 重複候補は商品マスタ ID でまとめる
- Decision: 商品名とひらがなの両方に一致した場合でも、同じ商品は 1 件だけ表示する。
- Rationale: 候補の重複は選択しにくさを生み、検索体験を悪化させる。
- Alternatives considered: 条件ごとに別表示。情報は増えるが、同一商品の重複が混乱を招く。

## Decision 5: 既存検索を壊さない
- Decision: ひらがなが未登録の商品でも、商品名検索では引き続き候補に表示する。
- Rationale: 過去データの即時更新を強制せず、段階的な移行を可能にする。
- Alternatives considered: ひらがな未登録商品を非表示にする案。既存運用が止まるため採用しない。
