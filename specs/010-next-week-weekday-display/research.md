# Research: 翌週の購入日を既定表示にする

## Decision 1: 既存の selectedWeekDateProvider を初期週の入口として使う

商品追加画面はすでに `selectedWeekDateProvider` を起点に週データを読み込んでいる。初期値だけを翌週へ寄せれば、ヘッダ表示・一覧読み込み・曜日選択が同じ state を共有したまま揃う。

## Decision 2: 翌週は「次の月曜始まりのカレンダー週」として扱う

このプロジェクトは週の開始を月曜日としているため、「翌週」は現在日から 7 日後の単純加算ではなく、次の月曜日から始まる 7 日間として定義する。これにより、月跨ぎや年跨ぎでも週表示と保存対象がぶれない。

## Decision 3: 週の算出は既存の週ユーティリティを流用する

`startOfWeek` / `endOfWeek` / `dateOnly` の既存ヘルパーをそのまま使えば、新しい日付ロジックを増やさずに翌週の範囲へ移せる。新しい独自計算を追加するよりも、既存の境界ルールに乗せる方が安全である。

## Decision 4: 永続化スキーマは変更しない

この変更は初期表示の基準を変えるだけであり、購入候補や週間明細の保存形式はそのままでよい。Drift の migration を増やす必要はない。

## Decision 5: テストは初期週の読み込みと UI 表示を確認する

画面起動時に翌週のスナップショットが読まれること、そして曜日切替 UI がその週に対応することを widget test で確認する。データ層の変更がないため、回帰検知は UI と state の結合点で十分である。

## Alternatives Considered

- 現在週のまま表示して、ユーザーに翌週を手動で選ばせる案: アプリの目的と矛盾するため不採用。
- 7 日後を単純に初期選択する案: 週境界とずれる可能性があるため不採用。
- 新しい週選択 state を追加する案: 既存の single source of truth を増やして複雑化するため不採用。
