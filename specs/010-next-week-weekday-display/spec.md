# Feature Specification: Next Week Weekday Display

**Feature Branch**: `010-next-week-weekday-display`  
**Created**: 2026-04-22  
**Status**: Draft  
**Input**: User description: "このアプリは、翌週の買い物をするアプリです。そのため、商品購入は、翌週の購入となるため、商品追加画面の曜日切り替えの週の表示は、翌週の日付を表示するようにする。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 商品追加時に翌週を表示する (Priority: P1)

ユーザーが商品追加画面を開いたとき、画面上部の曜日切替ヘッダは「翌週（次のカレンダー週）」の日付を表示する。

**Why this priority**: 本アプリは翌週の買い物を準備するために存在するため、商品追加操作が常に翌週を基準とすることが最重要である。

**Independent Test**: 商品追加画面を開き、表示されている曜日ヘッダの日付が「次週（月曜始まり）」の各日付と一致することを確認する。日付は端末ローカルのカレンダーに基づく。

**Acceptance Scenarios**:

1. **Given** ユーザーが任意の日（例: 2026-04-22）に商品追加画面を開いている、 **When** 画面が表示される、 **Then** ヘッダは次の週（月曜〜日曜）の各日付（例: 2026-04-27〜2026-05-03）を表示する。
2. **Given** 週末や月末を跨ぐ日付（例: 現在日が月末）でも、 **When** 画面表示時、 **Then** 表示される週は正しく翌週の日付を跨いで表示する（年・月を跨ぐケースも含む）。
3. **Given** ユーザーがヘッダ内の任意の曜日を選択すると、 **When** 決定操作を行う、 **Then** 選択された日の購入日は「翌週」の対応する日付として扱われる。

---

## Edge Cases

- 端末のローカルタイムゾーンが変わる（DST や手動変更）場合の表示整合性。
- 現在日が年末年始に近い場合の翌週算出（年跨ぎ）。
- ユーザーのロケールにより週開始日の定義が異なる可能性（本仕様では下記の前提を適用）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 商品追加画面の曜日ヘッダは、常に「翌週（次のカレンダー週）」の各日付を表示すること。
- **FR-002**: 各曜日の表示は端末のローカル日付基準に従い、日付はYYYY-MM-DDの内部表現で一意に決定できること（表示形式は UI に依存する）。
- **FR-003**: ユーザーが曜日を選択した場合、その選択は翌週の該当日付を対象とすること（保存・送信される購入予定日は翌週の日付であること）。
- **FR-004**: 表示と選択は月跨ぎ／年跨ぎケースでも一貫性があること。
- **FR-005**: ユーザーが期待どおり翌週を確認できるよう、視認性とアクセシビリティを確保すること（十分なコントラストとタップ領域）。

### Key Entities

- **WeekView**: 表示される一週間の範囲（週開始日、週終了日、各日の日付）
- **SelectedPurchaseDate**: ユーザーが商品追加で選択した購入予定日（必ず翌週内の日付）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% の受け入れテストケースで、ヘッダに表示される日付が「現在日を含む週の次のカレンダー週」と一致すること。
- **SC-002**: 主要な境界条件（月跨ぎ・年跨ぎ・タイムゾーン変化）で手動テスト項目がすべてパスすること。
- **SC-003**: UI 変更に関する自動テスト（ウィジェットテストまたは同等）が追加され、CI 上で合格すること。

## Assumptions

- 週の定義は「月曜日始まり」とする（既存プロジェクトの慣習に合わせる）。
- 表示のロケールは端末のロケールに従うが、内部的な日付算出はタイムゾーンを考慮して行われる。
- 既存の選択ロジック（ユーザーが選んだ曜日をどのように保存するか）は維持し、日付解決を翌週へリマップする。
