# Feature Specification: Category Order Settings

**Feature Branch**: `011-category-order`  
**Created**: 2026-04-22  
**Status**: Draft  
**Input**: User description: "商品カテゴリの並び順を変更する画面ですが、参照中のコンテキストから仕様を抽出してくさい。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 購入リストのカテゴリ順を並べ替える (Priority: P1)

ユーザーは設定画面でカテゴリの順序を見ながら、購入リスト画面に表示されるカテゴリ順をドラッグ操作で変更できる。

**Why this priority**: 購入リスト画面の見やすさは、日々の買い物操作のしやすさに直結するため、並び順の編集は最優先の機能である。

**Independent Test**: 設定画面でカテゴリを並べ替え、保存後に購入リスト画面を開いて表示順が変更されていることを確認する。

**Acceptance Scenarios**:

1. **Given** 複数のカテゴリが表示されている設定画面が開いている, **When** ユーザーがカテゴリをドラッグして位置を入れ替える, **Then** 画面上の順序が即座に変更される。
2. **Given** ユーザーが並べ替えを保存する, **When** 購入リスト画面を開く, **Then** カテゴリは保存した順序で表示される。
3. **Given** ユーザーが保存せずに画面を閉じる, **When** 設定画面を再度開く, **Then** 保存前の変更は反映されていない。

---

### User Story 2 - 並び順を既定に戻す (Priority: P2)

ユーザーは、変更した並び順を既定順に戻せる。

**Why this priority**: 変更後に元の状態へ戻せることは、安心して試せる編集体験に必要である。

**Independent Test**: 並び替え後にリセットを実行し、既定順に戻ることを確認する。

**Acceptance Scenarios**:

1. **Given** カテゴリ順が変更されている, **When** ユーザーがリセットを選ぶ, **Then** カテゴリ順は既定の順序に戻る。
2. **Given** リセット後に保存する, **When** 購入リスト画面を開く, **Then** 既定順が反映される。

---

### User Story 3 - 並び順変更を見やすく操作する (Priority: P3)

ユーザーは、並べ替え中の状態を見分けやすく、誤操作しにくい形で操作できる。

**Why this priority**: 頻繁な操作ではなくても、視認性と操作の分かりやすさは長期的な使いやすさに影響するため必要である。

**Independent Test**: 並べ替え対象の行が視覚的に分かり、ドラッグ対象が明確に示されることを確認する。

**Acceptance Scenarios**:

1. **Given** カテゴリ一覧が表示されている, **When** ユーザーが並べ替えを開始する, **Then** 変更対象の行が識別しやすく表示される。
2. **Given** スクリーンリーダー利用者が画面を操作する, **When** 並び替えに関する操作を行う, **Then** 操作の開始と完了が分かる。

---

## Edge Cases

- カテゴリ数が多い場合でも、一覧がスクロールしながら並べ替えできる。
- 保存前にアプリが閉じられた場合は、変更内容は失われる。
- 同じ `sort_order` が存在していても、保存時に整った順序へ再構成される。
- 購入リスト画面に表示されないカテゴリがある場合でも、表示対象の順序は崩れない。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ユーザーは設定画面で、購入リスト画面に表示されるカテゴリの順序を変更できなければならない。
- **FR-002**: カテゴリの並び順は、ドラッグ操作によって変更できなければならない。
- **FR-003**: 並び順の変更は、保存操作を行うまで永続化されてはならない。
- **FR-004**: 保存後、購入リスト画面は保存された順序でカテゴリを表示しなければならない。
- **FR-005**: ユーザーは変更前の既定順に戻せなければならない。
- **FR-006**: 並び替え操作中の行は、変更対象であることが分かるように視覚的に識別できなければならない。
- **FR-007**: 並び順の保存結果は、アプリを閉じて再起動しても維持されなければならない。
- **FR-008**: 並び順の保存は、カテゴリの表示順を制御する既存の仕組みを更新する形で行われなければならない。

### Key Entities

- **Category**: 購入リスト画面で表示されるカテゴリ。`id`、`name`、`sort_order`、`is_active` を持つ。
- **CategoryOrder**: 画面上で並び替えられるカテゴリ順の表現。保存時に `sort_order` として反映される。
- **PurchaseListDisplayOrder**: 購入リスト画面で実際に表示されるカテゴリの順序。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% の保存操作で、購入リスト画面のカテゴリ順が設定画面で保存した順序と一致する。
- **SC-002**: 変更の保存とリセットの主要フローを、ユーザーが 1 分以内に完了できる。
- **SC-003**: 並び替えの操作対象が明確であると、テスト参加者の少なくとも 90% が回答する。
- **SC-004**: 並び替えと保存の主要シナリオを確認する自動テストが追加され、継続的に合格する。

## Assumptions

- カテゴリの表示順は `sort_order` の昇順で決まる。
- 並び替え対象は、購入リスト画面で表示されるカテゴリに限定する。
- 既定順は、現行のカテゴリ順または `sort_order` の昇順として扱う。
- 変更はローカル保存される前提であり、クラウド同期は対象外とする。
